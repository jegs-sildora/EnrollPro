================================================================================
  ENROLLPRO — COMPLETE SYSTEM PROCESS OVERVIEW
  Junior High School (Grades 7–10) | DepEd-Aligned Enrollment Management System
================================================================================

Document purpose: Full end-to-end process reference covering Learner, Registrar,
and System Administrator workflows — from the start of a school year to its
completion and transition into the next school year.

Audience: Senior developers, QA engineers, school administrators, and EdTech
domain reviewers.

System timezone: Asia/Manila (all dates and timestamps are Manila-local).
Node runtime: 22 LTS. Backend: Express 5 + Prisma 6 + PostgreSQL.
Frontend: React + Vite. API base: /api.

1. USER ROLES AND ACCESS LEVELS
================================================================================

HEAD_REGISTRAR (also accepted as REGISTRAR in legacy contexts)
  - Full read/write access to all admission, enrollment, sectioning, EOSY,
    BOSY queue, export, and teacher management surfaces.
  - Cannot create or manage system users, cannot manage school year lifecycle
    transitions, cannot lock/unlock BOSY at the system level.
  - Cannot reopen a finalized EOSY section (admin-only).

SYSTEM_ADMIN
  - All HEAD_REGISTRAR permissions plus:
  - School year creation, rollover, status transitions (DRAFT → ARCHIVED).
  - BOSY system lock and unlock.
  - Curriculum and grade level management.
  - SCP program configuration (offering, steps, cutoff scores, grade requirements).
  - User account management (create, update, deactivate, reset password).
  - Profile lock on enrollment applications.
  - Reopen finalized EOSY sections.
  - Historical correction authorization override.

TEACHER
  - Read-only access to:
    - School year list (for SY switcher in the UI).
    - Grade levels.
    - Early registration applications list.
  - Cannot create, update, or delete any records.

LEARNER (portal — no JWT auth)
  - Access via portal PIN lookup only.
  - Can view own application status and enrollment information.
  - Can self-confirm BOSY return (beginning of new school year).
  - Cannot access any administrative surface.

PUBLIC (unauthenticated)
  - LRN existence check (GET /api/early-registrations/check-lrn/:lrn).
  - Early registration submission (POST /api/early-registrations).
  - ATLAS/integration read endpoints (GET /api/integration/v1/*).

2. SCHOOL YEAR LIFECYCLE (STATUS STATES)
================================================================================

School years progress through the following statuses. Only SYSTEM_ADMIN can
perform status transitions via POST /api/school-years/:id/transition.

  DRAFT
    The school year has been created (rolled over from the prior year) but is
    not yet visible to the public. Calendar dates, SCP configs, and sections can
    be configured while in this state.

  UPCOMING
    The school year is announced but not yet open for applications.
    Early registration window dates should be set before this state.

  PREPARATION
    Internal preparation phase. Staff-only access.

  ENROLLMENT_OPEN
    The early registration window (BEERF) and/or enrollment window (BEEF) are
    open. Portal control can be set to AUTO (opens based on date windows) or
    manually overridden.

  ACTIVE
    The school year is the currently active academic year. All learner and
    registrar operations are in full effect. Only one school year can be ACTIVE
    at a time; activating a new one automatically archives all others.

  BOSY_LOCKED
    Beginning-of-year sectioning and learner confirmation have been locked by
    System Admin. No further BOSY queue confirmations are permitted. Used to
    signal the start of formal classes.

  EOSY_PROCESSING
    End-of-year grade encoding is in progress. Sections are being finalized one
    by one.

  ARCHIVED
    The school year is complete and historical. All data is read-only
    (enforced by the historicalReadOnlyGuard middleware for all mutation
    requests scoped to non-active school years, unless a historical correction
    override token has been issued by System Admin).

Portal Control (SchoolYear.portalControl):
  AUTO     — Learner portal access opens and closes automatically based on
             earlyRegOpenDate/earlyRegCloseDate and enrollOpenDate/enrollCloseDate.
  OPEN     — Override: portal is forced open regardless of dates.
  CLOSED   — Override: portal is forced closed regardless of dates.

Key calendar date fields on SchoolYear:
  earlyRegOpenDate    — Phase 1 BEERF window opens
  earlyRegCloseDate   — Phase 1 BEERF window closes
  enrollOpenDate      — Phase 2 BEEF window opens
  enrollCloseDate     — Phase 2 BEEF window closes
  classOpeningDate    — First day of classes
  classEndDate        — Last day of classes

3. PRE-YEAR ADMINISTRATION (SYSTEM ADMIN)
================================================================================

Before the school year begins, the System Admin performs the following setup:

STEP 1 — CREATE OR ROLL OVER THE SCHOOL YEAR
  - First year: POST /api/school-years/activate
    Creates a new school year with yearLabel (e.g., "2025-2026"), sets it as
    ACTIVE, and ensures default grade levels exist (Grade 7–10).
  - Subsequent years: POST /api/school-years/rollover-draft
    Creates a DRAFT school year by cloning the structure (sections, SCP configs,
    section sort orders, homogeneous/snake flags) from the prior active year.
    The cloned year is stored in clonedFromId for BOSY rollover tracking.
  - Update draft details: PATCH /api/school-years/:id/rollover-draft
    Set yearLabel, all calendar date windows, sectioningConfig, and any
    enrollment gate parameters.

STEP 2 — SET CALENDAR DATES
  - PATCH /api/school-years/:id/dates
    Sets: classOpeningDate, classEndDate, earlyRegOpenDate, earlyRegCloseDate,
    enrollOpenDate, enrollCloseDate, sectionShiftWindowDays.

STEP 3 — TRANSITION TO PREPARATION / ENROLLMENT_OPEN
  - POST /api/school-years/:id/transition { status: "PREPARATION" }
  - POST /api/school-years/:id/transition { status: "ENROLLMENT_OPEN" }
    Once ENROLLMENT_OPEN, the portal control (AUTO mode) will begin accepting
    learner applications based on the date windows.

STEP 4 — ACTIVATE THE SCHOOL YEAR
  - POST /api/school-years/:id/transition { status: "ACTIVE" }
    Only one school year can be ACTIVE at a time. Previously ACTIVE years are
    automatically set to ARCHIVED.

4. CURRICULUM AND SCP CONFIGURATION (SYSTEM ADMIN)
================================================================================

GRADE LEVELS
  - GET /api/curriculum/:ayId/grade-levels
    Lists all grade levels for the school year (Grade 7, 8, 9, 10).
  - POST /api/curriculum/:ayId/grade-levels
    Creates a grade level (normally not needed — default grades are seeded on
    school year activation).
  - PUT /api/curriculum/grade-levels/:id
    Updates a grade level.
  - DELETE /api/curriculum/grade-levels/:id
    Removes a grade level (cascades to sections).

SPECIAL CURRICULAR PROGRAMS (SCP)
  - Programs: STE (Science, Technology, Engineering), ARTS (Special Program in
    the Arts), SPORTS, JOURNALISM, FOREIGN_LANGUAGE, TECH_VOC.
  - GET /api/curriculum/:ayId/scp-config
    Lists all SCP program configurations for the year.
  - PUT /api/curriculum/:ayId/scp-config
    Updates SCP configs in bulk, including:
    - isOffered (true/false) — whether the program is offered this year.
    - isTwoPhase (true/false) — whether screening has two stages (exam + interview).
    - cutoffScore — minimum score to pass the qualifying exam.
    - gradeRequirements — JSON: minimum grade per subject (e.g., Math >= 85, Science >= 85).
    - rankingFormula — JSON: weights for combining exam score and grade components.
    - ScpProgramStep[] — ordered assessment steps:
        * kind: QUALIFYING_EXAMINATION | PRELIMINARY_EXAMINATION |
                INTERVIEW | TALENT_AUDITION | PHYSICAL_FITNESS_TEST |
                SPORTS_SKILLS_TRYOUT | GENERAL_ADMISSION_TEST |
                APTITUDE_TEST | INTEREST_INVENTORY | SKILLS_ASSESSMENT |
                STANDARDIZED_ADMISSION_TOOL | FINAL_EXAMINATION
        * label, description, isRequired, cutoffScore, scheduledDate,
          scheduledTime, venue, rubric (JSON scoring rubric categories and
          criteria for interviews with maximum points per criterion).

SCP RANKINGS
  - GET /api/applications/scp-rankings
    Returns ordered applicant rankings based on the configured ranking formula,
    combining exam score and grade components for STE/ARTS/SPORTS programs.

================================================================================
5. SECTION SETUP (REGISTRAR / SYSTEM ADMIN)
================================================================================

Sections are per-school-year, per-grade-level containers for learners.

  - GET /api/sections?schoolYearId=&gradeLevelId=
    Lists all sections for the given school year and optional grade level filter.

  - POST /api/sections
    Creates a section:
      name         — Section display name (e.g., "Rizal", "Bonifacio")
      gradeLevelId — The grade level this section belongs to
      maxCapacity  — Maximum number of learners (default: 40)
      programType  — REGULAR | SCIENCE_TECHNOLOGY_AND_ENGINEERING |
                     SPECIAL_PROGRAM_IN_THE_ARTS | SPECIAL_PROGRAM_IN_SPORTS |
                     SPECIAL_PROGRAM_IN_JOURNALISM | SPECIAL_PROGRAM_IN_FOREIGN_LANGUAGE |
                     SPECIAL_PROGRAM_IN_TECHNICAL_VOCATIONAL_EDUCATION
      isHomogeneous — If true, batch algorithm groups learners by grade average.
      isSnake       — If true, distributes learners using snake-order ranking
                      across sections.
      tleSpecialization — For Grade 9/10 only (e.g., "HE - Cookery").
      sortOrder     — Display order in section lists.
      scpProgramConfigId — Links section to SCP program config (for SCP sections).

  - PUT /api/sections/:id  — Updates section details.
  - DELETE /api/sections/:id — Removes an empty section.

  Section Roster:
  - GET /api/sections/:id/roster
    Returns all learners officially assigned to the section (EnrollmentRecords
    linked to this sectionId).

6. TEACHER AND FACULTY MANAGEMENT (SYSTEM ADMIN)
================================================================================

Teachers are separate from User accounts. A Teacher record is a faculty profile;
User accounts are login credentials. Teachers can optionally be linked to a User
account via backfill.

  - GET /api/teachers — List all teachers.
  - POST /api/teachers — Create a teacher profile.
  - PUT /api/teachers/:id — Update teacher profile.
  - DELETE /api/teachers/:id — Remove teacher.

Teacher Designations (per school year):
  - TeacherDesignation links a teacher to a school year.
    Fields: isClassAdviser, isTic (teacher-in-charge), isTeachingExempt,
    advisoryEquivalentHoursPerWeek, advisorySectionId, ancillaryRoles[].
  - PATCH /api/teachers/:id/designation

Section Advisers:
  - GET /api/sections/teachers — Lists teachers eligible for advisory assignment.
  - Assigning an adviser creates a SectionAdviser record (historical ledger):
      status: ACTIVE | HANDED_OVER | REVOKED
      effectiveFrom, effectiveTo
  - Advisory handover: POST /api/sections/:id/handover-adviser
    Creates a new ACTIVE SectionAdviser and closes the prior one (HANDED_OVER).

User Account Management (SYSTEM_ADMIN only, at /api/admin/users):
  - GET /api/admin/users — List system users (REGISTRAR, SYSTEM_ADMIN, TEACHER).
  - POST /api/admin/users — Create user.
  - PUT /api/admin/users/:id — Update user.
  - PATCH /api/admin/users/:id/deactivate — Deactivate user.
  - PATCH /api/admin/users/:id/reactivate — Reactivate user.
  - PATCH /api/admin/users/:id/reset-password — Force password reset.

7. PHASE 1 — EARLY REGISTRATION (BEERF)
================================================================================

Phase 1 is the Basic Education Early Registration Form (BEERF) window.
Typically open January through May (before the school year formally starts).
Route prefix: /api/early-registrations

All DepEd Grade 7 new enrollees, transferees, and SCP applicants must pass
through Phase 1 to be eligible for Phase 2 enrollment.

────────────────────────────────────────────────────────────────────────────────
7a. LEARNER — ONLINE SELF-REGISTRATION
────────────────────────────────────────────────────────────────────────────────

  1. Learner visits the school portal.
  2. LRN check: GET /api/early-registrations/check-lrn/:lrn (public)
     - Returns whether the LRN already has an existing application for this
       school year. Prevents duplicate submissions.
     - If LRN is not found, learner may proceed without one (allowed only for
       incoming Grade 7 new enrollees and transferees — hasNoLrn flag).
  3. Learner fills out the BEERF form and submits:
     POST /api/early-registrations (public, no auth required)
     Payload includes:
       - Personal information: firstName, lastName, middleName, extensionName,
         birthdate, sex, placeOfBirth, religion, motherTongue.
       - LRN (or hasNoLrn=true for Grade 7 new enrollees without LRN).
       - Learner type: NEW_ENROLLEE | TRANSFEREE | RETURNING | CONTINUING | ALS | OSCYA
       - Applicant type: REGULAR | STE | ARTS | SPORTS | JOURNALISM |
                         FOREIGN_LANGUAGE | TECH_VOC
       - Grade level applying for (gradeLevelId).
       - Address (CURRENT and/or PERMANENT).
       - Guardian/parent information (MOTHER, FATHER, or GUARDIAN).
       - contactNumber, email.
       - reportedGrades (JSON: general average and individual subject grades
         from prior school — for SCP grade requirement screening).
       - SF9 certification letter status, originating school name.
       - isPrivacyConsentGiven: must be true.
  4. System creates:
     - A Learner record (if new) or matches existing Learner by LRN.
     - An EarlyRegistrationApplication with status = SUBMITTED_BEERF.
     - A unique trackingNumber (used to look up status later).
  5. Learner receives the trackingNumber for status tracking.

  No-LRN policy:
    Learners without an LRN (hasNoLrn) are allowed only for incoming Grade 7
    new enrollees and transferees. Learner.isPendingLrnCreation is set to true.
    These learners cannot be finalized into enrollment until a real LRN is
    assigned by DepEd and updated in the system.

────────────────────────────────────────────────────────────────────────────────
7b. REGISTRAR — F2F INTAKE (WALK-IN)
────────────────────────────────────────────────────────────────────────────────

  - POST /api/early-registrations/f2f
    Requires HEAD_REGISTRAR or SYSTEM_ADMIN.
    Same payload as the online form but submitted on behalf of the learner.
    Admission channel is recorded as F2F (face-to-face).

────────────────────────────────────────────────────────────────────────────────
7c. REGISTRAR — VERIFICATION AND DOCUMENT REVIEW
────────────────────────────────────────────────────────────────────────────────

  List applications:
  - GET /api/early-registrations?schoolYearId=&status=&search=&page=&limit=
    Filters: status, gradeLevelId, applicantType, learnerType, search (name/LRN).
    Teachers can view this list (read-only). Registrar and Admin have full access.

  View individual application:
  - GET /api/early-registrations/:id
  - GET /api/early-registrations/:id (show — full detail with assessment history,
    checklist, address, family members, reported grades, SCP results)

  Verify (SUBMITTED_BEERF → AWAITING_VERIFICATION or VERIFIED):
  - PATCH /api/early-registrations/:id/verify
    Registrar reviews submitted data, confirms accuracy.

  Document management:
  - POST /api/early-registrations/:id/documents (upload a document file)
  - DELETE /api/early-registrations/:id/documents (remove a document)
    Documents are stored as uploads on the server filesystem under /uploads.

  Checklist update:
  - PATCH /api/early-registrations/:id/checklist
    Records which physical documents have been presented and collected:
    - isPsaBirthCertPresented, isOriginalPsaBcCollected
    - isSf9Submitted, isSf10Requested
    - isGoodMoralPresented, isMedicalEvalSubmitted
    - isCertOfRecognitionPresented, isUndertakingSigned
    - isConfirmationSlipReceived, isSecondaryBirthDocPresented
    - academicStatus: PROMOTED | RETAINED | CONDITIONALLY_PROMOTED
    - isRemedialRequired (flags prior-year remedial completion requirement)

  Rejection:
  - PATCH /api/early-registrations/batch-process (with action: REJECT)
    Or individual reject with rejectionReason.

────────────────────────────────────────────────────────────────────────────────
7d. SCP APPLICANTS — ASSESSMENT PIPELINE
────────────────────────────────────────────────────────────────────────────────

  SCP applicants (STE, ARTS, SPORTS, etc.) go through one or two assessment
  phases defined in ScpProgramConfig steps.

  GRADE REQUIREMENT CHECK
    - Registrar verifies that the applicant's reportedGrades meet the
      ScpProgramConfig.gradeRequirements (e.g., Math ≥ 85, Science ≥ 85).
    - Applications not meeting grade requirements are moved to REJECTED or
      FOR_REVISION.

  ASSESSMENT SCHEDULING (Step 1 — Qualifying Exam or similar)
  - PATCH /api/early-registrations/:id/schedule-assessment
    Or batch: POST /api/early-registrations/batch/schedule-step
    Sets scheduledDate, scheduledTime, venue for the EarlyRegistrationAssessment
    linked to the corresponding ScpProgramStep.
    Status moves to: EXAM_SCHEDULED

  SCORE ENTRY
  - POST /api/early-registrations/batch/save-scores
    Records score and result for each applicant who took the exam.
    Status moves to: ASSESSMENT_TAKEN
    If score >= cutoffScore: moves toward next step or PASSED.
    If score < cutoffScore: moves to FAILED_ASSESSMENT.

  INTERVIEW (Step 2 — for isTwoPhase programs)
  - POST /api/early-registrations/batch/schedule-step
    Schedules the interview step.
    Status moves to: INTERVIEW_SCHEDULED
  - POST /api/early-registrations/batch/finalize-interview
    Records interview scores per rubric category/criterion.
    Status moves to: PASSED or FAILED_ASSESSMENT based on total.

  FINAL STATUS
    - PASSED: Applicant is eligible for Phase 2 enrollment.
    - FAILED_ASSESSMENT: Applicant does not proceed to enrollment for the SCP.
      They may be offered a REGULAR slot if available (registrar discretion).
    - READY_FOR_ENROLLMENT: Registrar has confirmed the applicant's eligibility
      and they are queued for Phase 2.

  SCP RANKINGS
  - GET /api/applications/scp-rankings
    After scores are entered, rankings are computed using the
    ScpProgramConfig.rankingFormula (weighted combination of exam scores,
    grades, and other components). Used to determine slot allocation.

────────────────────────────────────────────────────────────────────────────────
7e. REGISTRAR — BATCH OPERATIONS ON BEERF APPLICATIONS
────────────────────────────────────────────────────────────────────────────────

  Batch document verification:
  - POST /api/early-registrations/batch/verify-documents/preview
    Preview which applications will be affected.
  - POST /api/early-registrations/batch/verify-documents
    Marks documents as verified for selected applications in bulk.

  Batch assign regular section:
  - POST /api/early-registrations/batch/assign-regular-section/preview
  - POST /api/early-registrations/batch/assign-regular-section
    For REGULAR applicants who have passed verification, assigns them to a
    section in bulk.

  Batch status processing:
  - PATCH /api/early-registrations/batch-process
    Perform status transitions for multiple applications at once.

================================================================================
8. PHASE 2 — ENROLLMENT (BEEF)
================================================================================

Phase 2 is the Basic Education Enrollment Form (BEEF) window.
Typically open in June (beginning of the new school year).
Route prefix: /api/applications

Eligible applicants (from Phase 1 or continuing learners from prior year)
submit or have their Phase 2 BEEF form processed.

────────────────────────────────────────────────────────────────────────────────
8a. LEARNER — PORTAL STATUS TRACKING
────────────────────────────────────────────────────────────────────────────────

  Learner looks up their status using either:
    - Tracking number from their BEERF submission.
    - LRN + birthdate (PIN-based lookup).

  POST /api/learner/lookup
  Payload: { lrn, birthdate, pin } (pin is set during enrollment app creation)
  Response: Application status, section assignment, confirmation details.

  Learner sees one of the following status labels:
    SUBMITTED_BEERF        — Received, awaiting registrar review.
    AWAITING_VERIFICATION  — Under document review.
    VERIFIED               — Documents verified; ready for enrollment processing.
    ELIGIBLE               — Meets SCP grade requirements; proceeding to exam.
    EXAM_SCHEDULED         — Assessment date/venue confirmed.
    ASSESSMENT_TAKEN       — Exam completed; awaiting scoring.
    INTERVIEW_SCHEDULED    — Interview date/venue confirmed.
    PASSED                 — Cleared all assessments; eligible for enrollment.
    FAILED_ASSESSMENT      — Did not meet assessment cutoff.
    PENDING_BEEF           — Awaiting Phase 2 form completion.
    SUBMITTED_BEEF         — Phase 2 form submitted; awaiting registrar review.
    UNDER_REVIEW           — Registrar is reviewing Phase 2 application.
    FOR_REVISION           — Registrar has flagged for correction.
    READY_FOR_SECTIONING   — Approved and queued for section assignment.
    TEMPORARILY_ENROLLED   — Enrolled with pending document compliance.
    ENROLLED               — Officially sectioned and enrolled.
    OFFICIALLY_ENROLLED    — Final enrollment status (fully compliant).
    REJECTED               — Application denied.
    WITHDRAWN              — Learner withdrew application.

────────────────────────────────────────────────────────────────────────────────
8b. REGISTRAR — APPLICATION REVIEW AND APPROVAL
────────────────────────────────────────────────────────────────────────────────

  List Phase 2 applications:
  - GET /api/applications?schoolYearId=&status=&search=&gradeLevelId=&page=&limit=
    Returns all EnrollmentApplication records with flattened assessment data,
    including generalAverage resolved from:
      1. EnrollmentPreviousSchool.generalAverage (for new/transfer students)
      2. ApplicationChecklist.finalGeneralAverage
      3. reportedGrades.generalAverage (reported by student)
      4. EnrollmentRecord.finalAverage from the prior school year (EOSY grade)
      5. Learner.previousGenAve (SMART-synced grade)

  Verify application:
  - PATCH /api/applications/:id/verify
    Registrar confirms Phase 2 data is accurate. Status: VERIFIED.

  Approve application:
  - PATCH /api/applications/:id/approve
    Validates LRN requirements (learners with isPendingLrnCreation must have
    LRN assigned before final approval).
    Status moves to: READY_FOR_SECTIONING (or eligible next state).

  Update application:
  - PUT /api/applications/:id
    Edits application data fields (name, address, guardian info, etc.).
    Profile lock (isProfileLocked) must not be active.

  Profile lock (SYSTEM_ADMIN only):
  - PATCH /api/applications/:id/profile-lock
    Locks the application profile to prevent further edits.
    Used after official enrollment to protect data integrity.

  Reading profile assessment:
  - PATCH /api/applications/:id/reading-profile
    Sets readingProfileLevel: INDEPENDENT | INSTRUCTIONAL | FRUSTRATION | NON_READER
    Records readingProfileAssessedAt and readingProfileAssessedById.

  Enroll learner:
  - PATCH /api/applications/:id/enroll
    Creates an EnrollmentRecord with:
      sectionId (if already assigned), schoolYearId, learnerId, enrolledById.
    Status moves to: ENROLLED or OFFICIALLY_ENROLLED.

  Unenroll learner:
  - PATCH /api/applications/:id/unenroll
    Removes the enrollment record; returns to READY_FOR_SECTIONING or earlier.

  Special enrollment:
  - POST /api/applications/special-enrollment
    Handles non-standard enrollment cases (late enrollees, special admits).

  Batch processing:
  - PATCH /api/applications/batch-process
    Perform status transitions for multiple Phase 2 applications at once.

  Batch section assignment:
  - POST /api/applications/batch-assign-section
    Assigns a specific section to multiple applications at once.

────────────────────────────────────────────────────────────────────────────────
8c. REGISTRAR — DOCUMENT CHECKLIST AND READING PROFILE
────────────────────────────────────────────────────────────────────────────────

  Each EnrollmentApplication has an ApplicationChecklist:
    isPsaBirthCertPresented     — PSA birth certificate submitted
    isOriginalPsaBcCollected    — Original PSA BC taken by school
    isSf9Submitted              — SF9 Report Card submitted
    isSf10Requested             — SF10 Permanent Record requested
    isGoodMoralPresented        — Good Moral Certificate presented
    isMedicalEvalSubmitted      — Medical evaluation submitted
    isCertOfRecognitionPresented — Certificate of recognition
    isUndertakingSigned         — Undertaking/affidavit signed
    isConfirmationSlipReceived  — Confirmation slip given to learner
    isSecondaryBirthDocPresented — Secondary birth document (if no PSA)
    academicStatus: PROMOTED | RETAINED | CONDITIONALLY_PROMOTED
    isRemedialRequired          — Learner must complete remedial before enrolling

  Temporary enrollment:
    isTemporarilyEnrolled = true when a learner is conditionally enrolled
    pending document submission. Set complianceStatus to:
      PENDING | COMPLIED | OVERDUE
    documentaryDeadlineAt — deadline for document compliance.
    temporaryStatusDeadline — when the temporary status expires.

────────────────────────────────────────────────────────────────────────────────

9. SECTIONING PROCESS
================================================================================

Sectioning assigns ENROLLED learners (status: READY_FOR_SECTIONING or ENROLLED)
to specific sections. This happens after Phase 2 enrollment approvals.

────────────────────────────────────────────────────────────────────────────────
9a. BATCH SECTIONING ALGORITHM (REGISTRAR)
────────────────────────────────────────────────────────────────────────────────

  Prerequisites check:
  - GET /api/sections/batch-sectioning/prerequisites/:gradeLevelId
    Returns:
    - Number of ENROLLED/READY_FOR_SECTIONING learners.
    - Number of available section slots.
    - Whether slots are sufficient.
    - Prerequisite warnings or blockers.

  Run batch algorithm (preview — no commit):
  - POST /api/sections/batch-sectioning/run
    Payload (batchSectioningSchema):
      gradeLevelId    — Grade level to section.
      sectionIds      — List of section IDs to include in the run.
      isHomogeneous   — Group learners by general average (high → low).
      isSnake         — Use snake-order distribution (alternating high/low).
      prioritizeScp   — Assign SCP learners to SCP-designated sections first.
    Returns a preview: mapping of each learner to a proposed section.
    General average comes from the resolved generalAverage in flattenAssessmentData
    (prior-year EOSY finalAverage, SMART previousGenAve, or reported grade).

  Commit batch results:
  - POST /api/sections/batch-sectioning/commit
    Same payload as /run. Writes the section assignments to EnrollmentRecord.
    sectioningMethod is set to BATCH_ALGORITHM.
    dateSectioned is recorded.

────────────────────────────────────────────────────────────────────────────────
9b. INLINE SLOT ASSIGNMENT (REGISTRAR)
────────────────────────────────────────────────────────────────────────────────

  Unsectioned Learner Pool:
  - GET /api/sections/unsectioned-pool?schoolYearId=&gradeLevelId=
    Lists learners who are ENROLLED/OFFICIALLY_ENROLLED but do not yet have a
    section assignment. Includes:
      - promotionGenAve: prior-year finalAverage from EnrollmentRecord (if any),
        falling back to Learner.previousGenAve, then null.
      - applicantType, learnerType, lrn, full name.

  Inline assignment (from Unsectioned Pool to a specific section):
  - PATCH /api/sections/:sectionId/inline-slot
    Payload: { applicationId }
    Assigns the application to the section. Checks maxCapacity is not exceeded.
    sectioningMethod is set to MANUAL_OVERRIDE.
    Creates or updates the EnrollmentRecord.

────────────────────────────────────────────────────────────────────────────────
9c. LEARNER TRANSFER BETWEEN SECTIONS (REGISTRAR)
────────────────────────────────────────────────────────────────────────────────

  - PATCH /api/sections/:sectionId/transfer-learner
    Moves a learner from their current section to a new target section.
    Validates maxCapacity on the target section.
    Updates the EnrollmentRecord.sectionId.
    sf1Remarks may be set to explain the transfer.

────────────────────────────────────────────────────────────────────────────────
9d. SECTION ADVISER MANAGEMENT (REGISTRAR / ADMIN)
────────────────────────────────────────────────────────────────────────────────

  - GET /api/sections/teachers
    Lists teachers eligible to be advisers (linked to a TeacherDesignation for

11. END OF SCHOOL YEAR (EOSY) PROCESS
================================================================================

EOSY is the process of encoding final grades, setting promotion outcomes, and
locking section records at the end of the school year.

────────────────────────────────────────────────────────────────────────────────
11a. GRADE ENCODING PER SECTION (REGISTRAR)
────────────────────────────────────────────────────────────────────────────────

  List EOSY sections:
  - GET /api/eosy/sections?schoolYearId=
    Returns all sections for the school year with their finalization status
    (isEosyFinalized), enrolled count, and finalized count.

  Get section records:
  - GET /api/eosy/sections/:id/records
    Returns all EnrollmentRecords for the section with learner details:
      learnerId, lrn, firstName, lastName, sectionId, eosyStatus, finalAverage.

  Update individual record:
  - PATCH /api/eosy/records/:id
    Payload: { finalAverage: number, eosyStatus: EosyStatus }
    EosyStatus values:
      PROMOTED       — Learner passed all subjects; moves to next grade level.
      RETAINED       — Learner failed; repeats the same grade level.
      IRREGULAR      — Conditionally promoted; pending remedial subject completion.
                       (Internal status — maps to "Not Promoted" on DepEd SF5.)
      TRANSFERRED_OUT — Left the school during the year.
      DROPPED_OUT    — Dropped out during the year.
    finalAverage is stored as a Float (e.g., 85.50).

────────────────────────────────────────────────────────────────────────────────
11b. SECTION FINALIZATION (REGISTRAR)
────────────────────────────────────────────────────────────────────────────────

  Once all records in a section have been encoded:
  - POST /api/eosy/sections/:id/finalize
    Sets Section.isEosyFinalized = true.
    All EnrollmentRecords in the section are locked from further updates.
    This is a one-way operation — only SYSTEM_ADMIN can reopen via:
  - POST /api/eosy/sections/:id/reopen (SYSTEM_ADMIN only)

────────────────────────────────────────────────────────────────────────────────
11c. REMEDIAL RESOLUTION FOR IRREGULAR LEARNERS (REGISTRAR)
────────────────────────────────────────────────────────────────────────────────

  Learners with eosyStatus = IRREGULAR have been conditionally promoted and
  must complete remedial classes before their status is resolved.

  List pending remedial learners:
  - GET /api/remedial/pending
    Returns learners with IRREGULAR EOSY status who have not yet been resolved.

  Resolve remedial outcome:
  - PATCH /api/remedial/:learnerId/resolve
    Payload: { outcome: "PROMOTED" | "RETAINED" }
    Updates the EnrollmentRecord.eosyStatus to the final resolved value.
    If PROMOTED, the learner proceeds to the next grade in the BOSY rollover.
    If RETAINED, the learner repeats the same grade level.

  Note: IRREGULAR records are blockers for BOSY rollover. The system counts
  irregularBlockerCount in the BOSY readiness check. BOSY operations can
  begin but with a warning until all IRREGULAR cases are resolved.

────────────────────────────────────────────────────────────────────────────────
11d. SCHOOL-YEAR EOSY FINALIZATION (SYSTEM ADMIN)
────────────────────────────────────────────────────────────────────────────────

  Export lock check:
  - GET /api/eosy/school-year/:schoolYearId/export-lock
    Returns whether all sections have been finalized. Provides:
      totalSections, finalizedSections, canFinalizeSchoolYear, lockReason.
    All sections must be finalized before school-year-level finalization.

  School-year finalization:
  - POST /api/eosy/school-year/finalize
    Requires SYSTEM_ADMIN.
    Sets SchoolYear.isEosyFinalized = true.
    This is permanent — all mutation operations on the school year are globally
    locked after this point.
    A historicalReadOnlyGuard blocks all write operations to this school year.
    Only a Historical Correction Override (POST /api/admin/historical-correction/authorize)
    can temporarily unlock limited corrections for this school year.

────────────────────────────────────────────────────────────────────────────────
11e. LIS EXPORT (REGISTRAR / ADMIN)
────────────────────────────────────────────────────────────────────────────────

  Final LIS (Learner Information System) export:
  - GET /api/eosy/school-year/:schoolYearId/final-lis-export
    Downloads the finalized LIS CSV for submission to DepEd.
    Only available when all sections are finalized (export-lock check passes).

  SF1 export (class list / attendance register):
  - GET /api/export/sf1/:sectionId
    Exports the SF1 (School Form 1) for a specific section as a CSV.

  LIS Master export:
  - GET /api/export/lis-master
  - GET /api/applications/exports/lis-master
    Full learner data export across all sections.

  SF1 alternative export:
  - GET /api/applications/exports/sf1
    School-year-wide SF1 export.

12. SCHOOL YEAR TRANSITION (BOSY — BEGINNING OF SCHOOL YEAR)
================================================================================

BOSY (Beginning of School Year) is the transition process that rolls continuing
learners from the completed school year into the new school year.

────────────────────────────────────────────────────────────────────────────────
12a. ROLLOVER DRAFT CREATION (SYSTEM ADMIN)
────────────────────────────────────────────────────────────────────────────────

  - POST /api/school-years/rollover-draft
    Payload (rolloverSchoolYearSchema):
      yearLabel        — New school year label (e.g., "2026-2027").
      classOpeningDate — First day of classes.
      classEndDate     — Last day of classes.
      (and other calendar dates)
    What gets cloned from the source school year (clonedFromId):
      - Section names, maxCapacity, programType, sortOrder, isHomogeneous,
        isSnake, tleSpecialization, scpProgramConfigId.
      - SCP program configurations and steps.
      - Teacher designations.
    What does NOT get cloned:
      - Learner applications (generated fresh via BOSY sync).
      - Enrollment records (fresh per year).
      - Grade data.
      - Section advisers (must be reassigned each year).

  The rollover process also calls carryOverEligibleLearners which:
    - Reads all EnrollmentRecords from the source school year where the
      enrollment application status was ENROLLED, OFFICIALLY_ENROLLED, or
      TEMPORARILY_ENROLLED.
    - For each PROMOTED learner: creates a new EnrollmentApplication in the
      target year with:
        learnerType = CONTINUING
        status = PENDING_CONFIRMATION
        gradeLevelId = next grade level (Grade 7→8, 8→9, 9→10)
        intakeMethod = BATCH_CARRYOVER
      Grade 10 PROMOTED learners are skipped (JHS completers — they graduate).
    - For RETAINED learners: creates EnrollmentApplication at the same grade level.
    - Skips: DROPPED_OUT, TRANSFERRED_OUT, IRREGULAR (until resolved).
    - Skips: learners who already have an application in the target year.

────────────────────────────────────────────────────────────────────────────────
12b. BOSY SYNC — CARRYING OVER CONTINUING LEARNERS (REGISTRAR / ADMIN)
────────────────────────────────────────────────────────────────────────────────

  After the rollover draft is created, there may be learners whose prior-year
  applications were not captured during the initial rollover (e.g., added after
  the rollover ran). The BOSY sync fills these gaps.

  - POST /api/bosy/sync { schoolYearId }
    Auto-syncs: finds all EnrollmentRecords from the prior year (clonedFromId)
    for learners who do NOT yet have an EnrollmentApplication in the new year
    and are eligible (not DROPPED_OUT, not TRANSFERRED_OUT, not Grade 10 PROMOTED).
    Creates missing PENDING_CONFIRMATION applications.
    Returns { created: N } count of newly created applications.

  BOSY readiness check:
  - GET /api/bosy/readiness?schoolYearId=
    Auto-triggers a sync before returning. Returns:
      isEosyFinalized           — Whether prior year EOSY is complete.
      irregularBlockerCount     — Unresolved IRREGULAR learners from prior year.
      pendingConfirmationCount  — Learners still needing BOSY confirmation.
      readyForSectioningCount   — Confirmed learners queued for sections.
      enrolledCount             — Already sectioned continuing learners.
      jhsCompleterCount         — Grade 10 graduates (JHS completers).
      droppedCount              — Dropped learners in new year.

  BOSY queue:
  - GET /api/bosy/queue?schoolYearId=&status=&gradeLevelId=&search=&page=&limit=
    Lists all continuing learners and their current confirmation status.
    Each item shows: trackingNumber, status, lrn, name, gradeLevelName,
    academicStatus, priorSectionName, priorAdviserName.

────────────────────────────────────────────────────────────────────────────────
12c. LEARNER BOSY CONFIRMATION (LEARNER / REGISTRAR)
────────────────────────────────────────────────────────────────────────────────

  Each continuing learner must confirm their return to school for the new year.
  This moves them from PENDING_CONFIRMATION → READY_FOR_SECTIONING.

  Learner self-confirmation (no auth):
  - POST /api/learner/confirm-return
    Payload: { applicationId, confirmationConsent: true }
    Learner uses their portal PIN to confirm return.
    Status: PENDING_CONFIRMATION → READY_FOR_SECTIONING.

  Registrar confirmation (individual):
  - POST /api/bosy/confirm-return/:applicationId
    Registrar confirms on behalf of the learner (for walk-in or phone
    confirmation scenarios).

  Registrar bulk confirmation:
  - POST /api/bosy/bulk-confirm
    Payload: { applicationIds: number[] }
    Confirms multiple learners at once. Returns:
      confirmed: [ids that succeeded]
      failed: [{ id, reason }]

────────────────────────────────────────────────────────────────────────────────
12d. JHS COMPLETERS — GRADE 10 GRADUATES
────────────────────────────────────────────────────────────────────────────────

  Grade 10 learners with eosyStatus = PROMOTED are JHS completers.
  They are NOT rolled over into a new school year application.
  Learner.status is set to JHS_COMPLETER.

  - GET /api/bosy/completers?schoolYearId=
    Returns all JHS completers: learnerId, lrn, name, lastGradeLevel,
    lastYearEnrolled, lastSectionName.

  These learners are tracked in the system but receive no new enrollment
  application. Their records remain accessible for certificate/records purposes.

────────────────────────────────────────────────────────────────────────────────
12e. INCOMING GRADE 7 — NEW ENROLLEES AND TRANSFEREES
────────────────────────────────────────────────────────────────────────────────

  Incoming Grade 7 learners follow Phase 1 (BEERF) first, then Phase 2 (BEEF).
  They are NOT in the BOSY rollover queue (they have learnerType = NEW_ENROLLEE
  or TRANSFEREE, not CONTINUING).

  Early registration for Grade 7:
  - The BEERF window opens (per earlyRegOpenDate on the new school year).
  - Incoming Grade 7 learners (or their guardians) submit online.
  - Registrar processes them through the BEERF pipeline.
  - If applying for STE/ARTS/SPORTS: goes through the SCP assessment pipeline.
  - After PASSED, they proceed to Phase 2 BEEF during enrollOpenDate window.

  Transferees from other schools:
  - learnerType = TRANSFEREE.
  - Must present SF9/SF10 from their previous school.
  - generalAverage from EnrollmentPreviousSchool is used for SCP screening.
  - After verification and approval, proceed to Phase 2.

────────────────────────────────────────────────────────────────────────────────
12f. BOSY LOCK / UNLOCK (SYSTEM ADMIN)
────────────────────────────────────────────────────────────────────────────────

  BOSY lock signals the formal start of the school year and prevents further
  BOSY queue confirmations.

  - POST /api/admin/system/lock-bosy
    Sets SchoolYear.bosyLockedAt = now(), bosyLockedById = acting admin user.
    No further BOSY confirm-return operations are permitted.
    School year status transitions to BOSY_LOCKED.

  - POST /api/admin/system/unlock-bosy
    Removes the bosyLockedAt timestamp; re-enables BOSY confirmations.
    Used only if a late-arriving continuing learner needs to be processed.

13. NEW SCHOOL YEAR ACTIVATION
================================================================================

After the rollover draft has been configured and BOSY is underway:

  1. System Admin transitions the new school year to ACTIVE:
     POST /api/school-years/:id/transition { status: "ACTIVE" }
     - Previous ACTIVE school year is automatically set to ARCHIVED.
     - SchoolSetting.activeSchoolYearId is updated to the new year.
     - All system operations default to the new school year.

  2. Frontend context:
     - The client reads the active school year from GET /api/settings (public).
     - The response includes: activeSchoolYearId, schoolYearLabel, and portal
       control settings.
     - The SY Switcher (for REGISTRAR/ADMIN/TEACHER) allows viewing historical
       data by switching the school year context header (x-school-year-id).
     - Historical school years are read-only due to historicalReadOnlyGuard.

  3. Middleware school year resolution:
     - schoolYearContext middleware resolves the school year for each request:
       1. From x-school-year-id header (if provided by client).
       2. From SchoolSetting.activeSchoolYearId (default).
     - All route handlers use req.schoolYearId for data scoping.

18. APPLICATION STATUS REFERENCE
================================================================================

EARLY REGISTRATION (BEERF) STATUS FLOW:

  SUBMITTED_BEERF
    → Learner submitted the online or F2F BEERF form.
    → Registrar action: Review and verify.

  AWAITING_VERIFICATION
    → Registrar has received and is reviewing documents.

  VERIFIED
    → Documents confirmed accurate; applicant meets basic requirements.

  ELIGIBLE
    → Applicant meets SCP grade requirements; cleared to take assessment.

  EXAM_SCHEDULED
    → Qualifying exam or assessment date has been set and communicated.

  ASSESSMENT_TAKEN
    → Exam completed; awaiting score entry and result processing.

  INTERVIEW_SCHEDULED (two-phase SCP programs only)
    → Phase 2 interview has been scheduled.

  PASSED
    → All assessment requirements met; applicant is cleared for Phase 2 enrollment.

  FAILED_ASSESSMENT
    → Did not meet the cutoff score or interview threshold.
    → May be redirected to REGULAR lane if applicable.

  READY_FOR_ENROLLMENT
    → Registrar has confirmed eligibility; queued for Phase 2 BEEF processing.

  REJECTED
    → Application denied (documents not compliant, fails grade requirements, etc.).

  WITHDRAWN
    → Applicant withdrew voluntarily.

ENROLLMENT (BEEF) STATUS FLOW:

  SUBMITTED_BEEF / PENDING_BEEF
    → Phase 2 form received; awaiting registrar review.

  UNDER_REVIEW
    → Registrar is actively processing the enrollment application.

  FOR_REVISION
    → Registrar returned with corrections needed from the learner/guardian.

  VERIFIED
    → Phase 2 documents and data confirmed accurate.

  PENDING_CONFIRMATION
    → For continuing learners (BOSY queue): awaiting self-confirmation of return.

  READY_FOR_SECTIONING
    → Enrollment approved; learner queued for section assignment.

  TEMPORARILY_ENROLLED
    → Conditionally enrolled with pending documentary compliance.
    → complianceStatus tracks: PENDING → COMPLIED / OVERDUE.

  ENROLLED
    → Learner has been assigned to a section.
    → EnrollmentRecord created; sectionId is set.

  OFFICIALLY_ENROLLED
    → All conditions met; full enrollment confirmed.
    → Profile may be locked. Confirmation slip issued.

  REJECTED
    → Enrollment denied after Phase 2 review.

  WITHDRAWN
    → Learner withdrew after enrollment.

  TRANSFERRING_OUT
    → Transfer-out process initiated; awaiting documentation.

  TRANSFERRED_OUT
    → Official transfer complete. EnrollmentRecord.eosyStatus = TRANSFERRED_OUT.

  DROPPED
    → Learner dropped out. EnrollmentRecord.eosyStatus = DROPPED_OUT.

19. EOSY STATUS REFERENCE
================================================================================

Stored on EnrollmentRecord.eosyStatus. Set during end-of-year grade encoding.

  PROMOTED
    → Learner passed all subjects; advances to the next grade level in the
       following school year.
    → Grade 10 PROMOTED = JHS Completer (graduates from the system).

  RETAINED
    → Learner failed one or more subjects; repeats the same grade level.
    → BOSY rollover creates an application at the same grade level.

  IRREGULAR
    → Conditionally promoted. Learner must complete remedial subject(s).
    → Internal system status only; maps to "Not Promoted" on DepEd SF5.
    → Blocks clean BOSY rollover until resolved via /api/remedial/:learnerId/resolve.
    → Resolution options: PROMOTED (if remedial passed) or RETAINED (if failed).

  TRANSFERRED_OUT
    → Learner transferred to another school during or at the end of the year.
    → Excluded from BOSY rollover.

  DROPPED_OUT
    → Learner dropped out during the year.
    → Excluded from BOSY rollover.
